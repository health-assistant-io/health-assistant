# Changelog

All notable changes to Health Assistant are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

> **Workflow:** every user-facing change adds a line under `## [Unreleased]` at
> commit time (see [docs/RELEASE_PROCESS.md](docs/RELEASE_PROCESS.md)). At release
> time, `## [Unreleased]` is renamed to `## [vX.Y.Z] - YYYY-MM-DD` and a fresh
> `## [Unreleased]` is opened above it.

- **Fix (documents): PDF / image preview no longer blocked by `X-Frame-Options: DENY`.** The document preview (`AuthenticatedPdf` / `PdfViewer`, rendered as a same-origin `<iframe src="/api/v1/documents/{id}/download?token=...">`) was rejected by the browser because `security_headers_middleware` (`app/main.py`) stamped `X-Frame-Options: DENY` on **every** response unconditionally, including the inline download the preview embeds. The middleware now uses `setdefault` so the `DENY` baseline is preserved for all routes while a route that legitimately needs to be framed can opt out. The document download endpoint (`GET /api/v1/documents/{id}/download`) sets `X-Frame-Options: SAMEORIGIN` **only when serving a file inline** (`should_serve_inline` is `True` — PDFs, images, text; active-content types like svg/html/xml/js stay DENY and are forced to `attachment` anyway, so no clickjacking surface is opened). New regression test asserts the middleware honors an explicit per-route override without dropping the other baseline headers. Audit A7 test still green (the `DENY` default is intact on a normal route).

- **AI chat: voice input (speech-to-text) + capability-based model features.** Two coupled changes that rework how models declare what they can do and add hands-free chat input. **(1) Capability-based models:** replaced the implicit "every model is a chat model" assumption with an explicit capability set. A new `AIModelCapability` enum (`text` / `vision` / `audio_input`) is stored as a JSONB `capabilities` array on `AIModel` (migration `s1t2m3o4d5e6`, default `["text"]` so existing models keep their meaning). A single source-of-truth mapping (`app/ai/providers/capabilities.py` `required_capabilities_for_task`) declares what each task needs — `chat`→text, `ocr`→vision, `transcription`→audio_input, every other text-generation task→text — so the AI-config Models UI now shows **feature toggles** (Text is always on; Vision / Audio Input are opt-in chips with icons + colored badges on the list row). A model can be multimodal (e.g. `gpt-4o` = text+vision) or purpose-built (`whisper-1` = audio_input). The `transcription` task type was added to `TaskType`, surfaced in the config summary (`AIConfigSummary.transcription`) + a new `voice_input` workflow, so admins can create a `transcription` task assignment pointing at an `audio_input` model. **(2) Voice input:** a new `POST /ai-assistance/transcribe` endpoint accepts a compressed audio upload, resolves the `transcription` assignment via a new `AIProviderService.get_stt_target` (validates the resolved model advertises `audio_input`; rejects a misconfigured chat-only assignment with a clear guard message), and POSTs it to the provider's `/audio/transcriptions` (OpenAI-compatible) returning `{text}`. Audio is **ephemeral** — never persisted (it may contain PHI); the resulting text is prompt-guard-scanned for audit correlation and then placed in the chat input box for the user to review/edit before sending (it then flows through the normal chat pipeline). Size (20 MiB) + MIME guards at the trust boundary; backend `app/ai/assistance/stt.py` (`STTTarget`, `TranscriptionError`, `transcribe_audio`) + new config (`AI_STT_MAX_AUDIO_BYTES`, `AI_STT_TIMEOUT_SECONDS`, `OPENAI_STT_MODEL`). **Frontend:** `useVoiceRecorder` hook records Opus/16 kHz mono/24 kbps via `MediaRecorder` (~90 KB / 30s — tiny uploads) with a 60s hard cap; `ChatVoiceButton` is a **single unified control** handling both interaction models by gesture — tap (down→up < 250 ms) = toggle (tap to start, tap to stop), press & hold = push-to-talk — with a live elapsed timer and a clear "unsupported/permission-denied" path. Transcribed text appends into the editable input box (preserves typed draft + focuses for review). i18n added (en + el). 12 backend tests (`test_stt_capabilities.py`) cover the capability mapping, baseline inclusion, and STT-target resolution/enforcement; the squash-schema test was relaxed to allow a baseline + incremental follow-ups. Frontend tsc + eslint + build green.

- **5 of 8 advertised FHIR Condition search params silently no-op'd.** `RESOURCE_PARAMS['Condition']` advertised `clinical-status`, `code`, `category`, `verification-status`, and `encounter`, but the generic dispatcher had no real handler for them against `ClinicalEvent` (the projection model): `clinical-status` mapped to a nonexistent `clinical_status` column, `code` ran the JSONB containment path against a `String` column, `category`/`encounter` need joins the dispatcher can't infer, and `verification-status` has no equivalent at all. ### Fixed
- **Celery periodic tasks crashed intermittently with `RuntimeError: Event loop is closed` / `Future attached to a different loop`.** Root cause: each task's `get_async_session()` created a fresh `AsyncEngine` with its own connection pool, and the `@async_task` decorator created a fresh `asyncio` event loop per task. asyncpg connection protocol transports bound to a previous task's loop resurfaced in subsequent tasks via SQLAlchemy's pool, hitting a closed loop. Fix: the engine is now a worker-scoped singleton using `poolclass=NullPool` so no DB connection is ever reused across loops; engine disposal is wired to Celery's `worker_process_shutdown` signal rather than per-task. Live production outage observed on `check_notification_triggers` (2026-06-27).
- **`relative_score` was never computed for OCR-sourced observations.** Every OCR-sourced observation had `relative_score = NULL`, silently disabling the biomarker engine's scoring feature (`_get_observation_status` step 2 in `analytics_service.py` was dead code for OCR data; everything fell through to the hardcoded `STANDARD_RANGES` dict). `save_observation` (in `app/ai/pipeline/persistence.py`) now computes it using the same formula as the integrations SDK's `ObservationBuilder.build()`: `(value - ref_min) / (ref_max - ref_min)` clamped to `[0.0, 1.0]` when both bounds are present with `max > min`; incomplete range (only one bound) → `0.5`; no range → `NULL`.
- **`normalized_value` had two bugs in the OCR persistence path.** (1) Skip-condition: it was only computed when the biomarker had a preferred unit *different* from the matched raw unit; auto-created biomarkers (no preferred unit) kept raw_value, mixing units across labs in trend history. (2) Wrong direction: `val_float * matched_unit.conversion_multiplier` converts raw → base SI, but the result was labelled as-preferred. Fixed: `normalized_value` is now always expressed in the biomarker's preferred unit (`raw → base → preferred`), falling back to base SI when no preferred unit is set so trends stay consistent across labs.
- **`MedicationCatalog` / `AllergyCatalog` dedup used substring matching.** The uniqueness check `Catalog.name.ilike(name)` treated `_` and `%` as wildcards, so `"Vitamin_B12"` would silently match an existing `"Vitamin-B12"` (underscore matches any single char) and skip the insert. Switched all 5 dedup sites (`ai/pipeline/ontology.py`, `seed_service.py` ×2, `import_service.py` ×2) to `func.lower(Catalog.name) == func.lower(name)` for strict case-insensitive exact match. Catalog *search* features (legitimate substring `ilike('%q%')`) are unchanged.
- **Cumulative extraction was triggered even when all OCR documents produced empty text** (blank pages, OCR failures). The else branch of `_check_trigger_cumulative` fired `cumulative_extraction.delay(...)` "anyway to handle metadata if needed", wasting an LLM call and producing hallucinated or empty impressions. The else branch now marks the exam completed (`extraction_status="completed"`, `impressions=""`) and skips the LLM call entirely.
- **Web Push dead subscriptions accumulated forever.** `webpush_service.send_web_push` swallowed every `WebPushException` including the 410 Gone / 404 Not Found cases that RFC 8030 specifies as "subscription permanently dead", returning `False`. The caller couldn't distinguish transient failure from permanent death, so dead rows in `notification_subscriptions` stayed `is_active=True` and every PUSH cycle re-attempted them. `send_web_push` now raises a new `SubscriptionExpired` exception on 410/404; `deliver_notification`'s PUSH loop catches it and marks the matching `NotificationSubscription.is_active = False`.
- **`doctors.py` list endpoint leaked internal exception text.** The handler was wrapped in a `try/except Exception` that did `print(...)` + `traceback.print_exc()` + re-raised as `HTTPException(500, detail=str(e))`, bypassing the global exception handler. The wrapper is removed; the global handler now logs with a correlation id and returns a sanitized 500.
- **`check_observation_access` raised `AttributeError` → HTTP 500** when an observation's `subject` JSONB dict had no `"reference"` key (`.get()` returned `None`, then `None.split()` raised). Now defensively coerces via `(observation.subject or {}).get("reference", "")` so an absent reference yields `""` and the downstream surfaces a clean 400.
- **Anatomy tests failed on second run** (`UniqueViolationError` on `anatomy_structures.slug`). `anatomy_structures.slug` has a globally-unique constraint (anatomy is an ontology — heart is heart in every tenant), but tests used stable slugs like `"test-heart"` and the suite has no per-test transaction rollback. Rows accumulated across runs in different tenants created by the per-call `system_admin_headers` fixture. Added an autouse module-level fixture that deletes from `anatomy_structures` before each test (CASCADE handles `anatomy_relations` + `event_anatomy_links`). The tables aren't seeded by migrations — the only data in them during tests IS test data — so truncating is safe.
- **"Create New" dropdown hidden behind the sidebar.** The CreateMenu popup (New Examination / New Medication / etc.) is rendered via a portal at `document.body`, but its `z-index` (200) was lower than the sidebar container's (`z-[950]` in `Layout.tsx`). Since the layout root doesn't establish its own stacking context, the sidebar painted over the portaled dropdown. Bumped the dropdown to `z-[1000]` so it now correctly floats above the navigation sidebar.
- **Anatomy SVG assets failed to open in XML editors** with "XML declaration allowed only at the start of the document" and undeclared namespace-prefix errors. The `<?xml ?>` declaration was moved above the license comment, and orphaned Inkscape/RDF metadata blocks (using stripped `sodipodi:`/`rdf:`/`cc:`/`dc:` prefixes) were removed. Both files now validate as well-formed XML with zero errors.

### Fixed
- **Notification periodic task crashed with "Future attached to a different loop".** `check_notification_triggers` delegated to `NotificationManager.process_due_triggers()`, which opened its own session via the global pooled app engine (`AsyncSessionLocal`, `pool_pre_ping=True`). In a Celery prefork worker that engine's asyncpg connections are bound to the master's (already-closed) event loop, so the pool's pre-ping checkout raised `RuntimeError: Future ... attached to a different loop` and teardown raised `RuntimeError: Event loop is closed` — scheduled/recurring medication reminders intermittently failed to fire. The task now owns a worker-scoped `NullPool` session (via `get_async_session()`, the same pattern as `sync_active_integrations`) and injects it into `process_due_triggers(session=...)` / `fire_notification(session=...)`. A failed trigger now rolls back the shared session so the remaining triggers in the cycle still process. Request-path callers are unchanged (they fall back to the app engine).
