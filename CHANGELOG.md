# Changelog

All notable changes to Health Assistant are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

> **Workflow:** every user-facing change adds a line under `## [Unreleased]` at
> commit time (see [docs/RELEASE_PROCESS.md](docs/RELEASE_PROCESS.md)). At release
> time, `## [Unreleased]` is renamed to `## [vX.Y.Z] - YYYY-MM-DD` and a fresh
> `## [Unreleased]` is opened above it.

## [Unreleased]

- **Sidebar nav restructured: Patient Record/Clinical Data merged, Catalogs flattened, Notifications promoted.** The main sidebar had a conceptually muddy boundary ŌĆö "Patient Record" (9 items) and "Clinical Data" (2 items) overlapped (biomarkers/medications are clinical data; examinations/documents are patient record), and "Patient Record" was an unstructured grab-bag mixing identity, analytics, treatments, alerts, and time-views. Changes: (1) **Clinical Data is merged into Patient Record**, now organised with internal `section` dividers (the same mechanism the Administration group already used) ŌĆö *Clinical Record* (Examinations, Documents, Biomarkers, Correlative Analytics), *Treatments & Alerts* (Medications, Vaccinations, Allergy Alerts), *Timeline* (Clinical Events, Clinical Calendar) ŌĆö so the 10-item group is scannable instead of a flat list. (2) **Notifications promoted to a top-level item** ŌĆö it is app-level (a header bell already exists), not patient-scoped, so it no longer sits under Patient Record. (3) **Catalogs collapsed from a 6-item expanding group to a single flat link** ŌĆö `/catalogs` already opens a unified tabbed workspace; the six `?type=` sub-items were redundant shortcuts that added nav noise. Result: 5 top-level items (Dashboard, Patient Record, Notifications, Catalogs, AI Assistant) + role-gated Administration, down from 6, with the mega-group made navigable via sections. New i18n keys (`section_clinical_record`/`section_treatments_alerts`/`section_timeline`) added to en + el; `clinical_data` key retained for compatibility.

- **Catalog: unified filter UI (all dropdown FacetChips) + category tag on every row + selected-value pills.** Filters were applied inconsistently across catalog types ŌĆö `concept` (kind) and `anatomy` (class) used inline toggle chips in the toolbar while `biomarker`/`allergy`/`vaccine`/`medication` used popover-dropdown `FacetChip`s in a second toolbar row. Both are now **server-side `FacetChip` dropdowns** (`conceptKindFacet` with static `ConceptKind` enum options; `anatomy` class facet with fetched `anatomy_class` options via a `makeClassFacet` factory in `catalogFacetRegistry`), so every catalog type uses the same dropdown component and filtering composes correctly with pagination (server-side, before `limit`/`offset`) rather than only over the loaded 50-item page. The previously-dead `catalogFilter.serverParams` (computed by `useFilterState` but never consumed) is now wired into the `load()`/`loadMore()` API calls, with a stringified `serverFilterKey` dependency so client-side facet toggles don't trigger a refetch. Row tags were equally uneven ŌĆö only `biomarker` and `anatomy` surfaced their category as a colored `ClassBadge`; `medication`/`allergy`/`vaccine`/`concept` showed only the scope badge. The three FHIR-catalog `to_dict()` methods now emit `class_concept_slug` + `class_concept_name` (the `class_concept` relationship is already `lazy="selectin"`, so no N+1) and concepts now surface `primary_kind` as a colored `KIND_COLORS` badge (concepts carry no class FK). The generic `ClassBadge` in `CatalogBrowser` falls back to `primary_kind` when no `class_concept` is present, and its click-to-filter now routes to the matching facet (kind for concepts, class otherwise). Finally, the removable **active-value pills** built into `FacetChip` (`showActivePills`) are now enabled in both the catalog workspace and the picker browse modal, so selected filter values render as removable chips next to each dropdown without having to open it (desktop-only; mobile sees the count summary on the trigger). Tests updated (facet registry + facet defs); `CatalogToolbar` lost the inline-chip props/render. The picker modal filters out server-mode facets (kind/class) since it filters in-memory and doesn't refetch.

### Fixed
- **Catalog item edits not persisted + preview not refreshing.** Editing a catalog item (e.g. a medication) silently failed: the PUT returned a 500 (`property 'is_custom' of 'MedicationCatalog' object has no setter`) because the adapter's `_writable()` filter used `hasattr`, which admitted the computed `is_custom` `@property` (derived from `tenant_id`) ŌĆö and relationships ŌĆö through to `setattr`. `_writable` now restricts to real SQLAlchemy mapper columns (`inspect(model).columns`), so computed properties/relationships are excluded from mass-assignment across every catalog. On the frontend the failure was additionally hidden by an **empty `catch { /* toast */ }`** in `handleSave`/`handleDelete` (`CatalogWorkspace`) ŌĆö the error was swallowed with no toast, so edits looked like they did nothing while relations (which the inline `CatalogRelationsEditor` persists immediately via its own edge calls) appeared to update. Both handlers now surface the backend `detail` via `toast.error`, show a success toast, patch the returned row straight into the local item list (so the Info preview reflects the edit instantly), disable the Save button while in-flight ("SavingŌĆ”"), and bump a `relationsRevision` counter that drives a new `refreshKey` prop on `CatalogRelationsGraph` so the Relations graph refetches on save (it previously fetched once per item selection and stayed stale until re-selected, without losing the depth/view/node-selection state). `npm run build` (tsc strict) + `npm run lint` clean; 122 catalog tests green.

- **Catalog type dropdown chevron + relations graph fill.** (1) The catalog-type select's chevron sat immediately after the type name instead of at the right edge ŌĆö the trigger's label span is now `flex-1` so the arrow is pushed to the far right of the button (standard dropdown affordance). (2) The Relations tab graph was locked to a fixed `h-[500px]` and didn't fill the preview pane ŌĆö `CatalogRelationsGraph` now uses a `flex flex-col h-full` root with the graph canvas as `flex-1 min-h-[400px]` (and the cards view in a `flex-1` scroll area), so it expands to fill all available width and height of the preview pane instead of leaving dead space.

- **Catalog selection contrast + preview header.** (1) The active-row left accent only appeared on the first row because the `border-l-4`/`border-transparent` approach collided with the list's `divide-y` borders ŌĆö switched the accent to a `box-shadow inset` so it renders on every selected row, and softened the selection fill (`bg-blue-100`ŌåÆ`bg-blue-50`, dark `/50`ŌåÆ`/30`; card `ring-2`ŌåÆ`ring-1 ring-inset`). ### Fixed
- **Celery periodic tasks crashed intermittently with `RuntimeError: Event loop is closed` / `Future attached to a different loop`.** Root cause: each task's `get_async_session()` created a fresh `AsyncEngine` with its own connection pool, and the `@async_task` decorator created a fresh `asyncio` event loop per task. asyncpg connection protocol transports bound to a previous task's loop resurfaced in subsequent tasks via SQLAlchemy's pool, hitting a closed loop. Fix: the engine is now a worker-scoped singleton using `poolclass=NullPool` so no DB connection is ever reused across loops; engine disposal is wired to Celery's `worker_process_shutdown` signal rather than per-task. Live production outage observed on `check_notification_triggers` (2026-06-27).
- **`relative_score` was never computed for OCR-sourced observations.** Every OCR-sourced observation had `relative_score = NULL`, silently disabling the biomarker engine's scoring feature (`_get_observation_status` step 2 in `analytics_service.py` was dead code for OCR data; everything fell through to the hardcoded `STANDARD_RANGES` dict). `save_observation` (in `app/ai/pipeline/persistence.py`) now computes it using the same formula as the integrations SDK's `ObservationBuilder.build()`: `(value - ref_min) / (ref_max - ref_min)` clamped to `[0.0, 1.0]` when both bounds are present with `max > min`; incomplete range (only one bound) ŌåÆ `0.5`; no range ŌåÆ `NULL`.
- **`normalized_value` had two bugs in the OCR persistence path.** (1) Skip-condition: it was only computed when the biomarker had a preferred unit *different* from the matched raw unit; auto-created biomarkers (no preferred unit) kept raw_value, mixing units across labs in trend history. (2) Wrong direction: `val_float * matched_unit.conversion_multiplier` converts raw ŌåÆ base SI, but the result was labelled as-preferred. Fixed: `normalized_value` is now always expressed in the biomarker's preferred unit (`raw ŌåÆ base ŌåÆ preferred`), falling back to base SI when no preferred unit is set so trends stay consistent across labs.
- **`MedicationCatalog` / `AllergyCatalog` dedup used substring matching.** The uniqueness check `Catalog.name.ilike(name)` treated `_` and `%` as wildcards, so `"Vitamin_B12"` would silently match an existing `"Vitamin-B12"` (underscore matches any single char) and skip the insert. Switched all 5 dedup sites (`ai/pipeline/ontology.py`, `seed_service.py` ├Ś2, `import_service.py` ├Ś2) to `func.lower(Catalog.name) == func.lower(name)` for strict case-insensitive exact match. Catalog *search* features (legitimate substring `ilike('%q%')`) are unchanged.
- **Cumulative extraction was triggered even when all OCR documents produced empty text** (blank pages, OCR failures). The else branch of `_check_trigger_cumulative` fired `cumulative_extraction.delay(...)` "anyway to handle metadata if needed", wasting an LLM call and producing hallucinated or empty impressions. The else branch now marks the exam completed (`extraction_status="completed"`, `impressions=""`) and skips the LLM call entirely.
- **Web Push dead subscriptions accumulated forever.** `webpush_service.send_web_push` swallowed every `WebPushException` including the 410 Gone / 404 Not Found cases that RFC 8030 specifies as "subscription permanently dead", returning `False`. The caller couldn't distinguish transient failure from permanent death, so dead rows in `notification_subscriptions` stayed `is_active=True` and every PUSH cycle re-attempted them. `send_web_push` now raises a new `SubscriptionExpired` exception on 410/404; `deliver_notification`'s PUSH loop catches it and marks the matching `NotificationSubscription.is_active = False`.
- **`doctors.py` list endpoint leaked internal exception text.** The handler was wrapped in a `try/except Exception` that did `print(...)` + `traceback.print_exc()` + re-raised as `HTTPException(500, detail=str(e))`, bypassing the global exception handler. The wrapper is removed; the global handler now logs with a correlation id and returns a sanitized 500.
- **`check_observation_access` raised `AttributeError` ŌåÆ HTTP 500** when an observation's `subject` JSONB dict had no `"reference"` key (`.get()` returned `None`, then `None.split()` raised). Now defensively coerces via `(observation.subject or {}).get("reference", "")` so an absent reference yields `""` and the downstream surfaces a clean 400.
- **Anatomy tests failed on second run** (`UniqueViolationError` on `anatomy_structures.slug`). `anatomy_structures.slug` has a globally-unique constraint (anatomy is an ontology ŌĆö heart is heart in every tenant), but tests used stable slugs like `"test-heart"` and the suite has no per-test transaction rollback. Rows accumulated across runs in different tenants created by the per-call `system_admin_headers` fixture. Added an autouse module-level fixture that deletes from `anatomy_structures` before each test (CASCADE handles `anatomy_relations` + `event_anatomy_links`). The tables aren't seeded by migrations ŌĆö the only data in them during tests IS test data ŌĆö so truncating is safe.
- **"Create New" dropdown hidden behind the sidebar.** The CreateMenu popup (New Examination / New Medication / etc.) is rendered via a portal at `document.body`, but its `z-index` (200) was lower than the sidebar container's (`z-[950]` in `Layout.tsx`). Since the layout root doesn't establish its own stacking context, the sidebar painted over the portaled dropdown. Bumped the dropdown to `z-[1000]` so it now correctly floats above the navigation sidebar.
- **Anatomy SVG assets failed to open in XML editors** with "XML declaration allowed only at the start of the document" and undeclared namespace-prefix errors. The `<?xml ?>` declaration was moved above the license comment, and orphaned Inkscape/RDF metadata blocks (using stripped `sodipodi:`/`rdf:`/`cc:`/`dc:` prefixes) were removed. Both files now validate as well-formed XML with zero errors.

### Fixed
- **Notification periodic task crashed with "Future attached to a different loop".** `check_notification_triggers` delegated to `NotificationManager.process_due_triggers()`, which opened its own session via the global pooled app engine (`AsyncSessionLocal`, `pool_pre_ping=True`). In a Celery prefork worker that engine's asyncpg connections are bound to the master's (already-closed) event loop, so the pool's pre-ping checkout raised `RuntimeError: Future ... attached to a different loop` and teardown raised `RuntimeError: Event loop is closed` ŌĆö scheduled/recurring medication reminders intermittently failed to fire.
