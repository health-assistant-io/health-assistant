# Changelog

All notable changes to Health Assistant are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

> **Workflow:** every user-facing change adds a line under `## [Unreleased]` at
> commit time (see [docs/RELEASE_PROCESS.md](docs/RELEASE_PROCESS.md)). At release
> time, `## [Unreleased]` is renamed to `## [vX.Y.Z] - YYYY-MM-DD` and a fresh
> `## [Unreleased]` is opened above it.

## [Unreleased]

### Added
- **Download the instance's seeds as a ZIP (UI + dev unpack helper).** A maintainer can now pull the curated taxonomy/anatomy/catalog off a running instance (including one on a different machine) without SSH: a "Seeds" download button on the **TaxonomyManager** page (`/admin/system/taxonomy`, SYSTEM_ADMIN-only) calls `GET /admin/seeds/export.zip` and saves a ZIP of the 9 seed-format JSON files. The download is read-only — the server never writes its own `data/seeds/`. The maintainer transfers the ZIP to their dev machine and runs `python scripts/unpack_seeds_zip.py <zip>` (backs up existing `data/seeds/*.json` to `.backup-<timestamp>/`, refuses unknown/path-traversal entries, then extracts), then reviews with `git diff data/seeds/`. Frontend `seedService.downloadSeedsZip()` mirrors the existing backup-download pattern.
- **Export an instance's data back into the shipped seeds (`SeedExportService` + CLI).** A curator can now build the canonical taxonomy/anatomy/biomarker/catalog set in a running instance via the UI + AI, then snapshot it into `backend/data/seeds/*.json` to replace the hardcoded seeds. New `app/services/seed_export_service.py` is the strict inverse of `SeedService` — one method per seed file — emitting the slug-keyed format (slug is the stable join key; `coding_system`+`code` carried as identity *evidence*, not the key). The CLI `python scripts/export_seeds.py` writes the files through a safety pipeline (stage → back up existing to `.backup-<timestamp>/` → atomic write), so the result is reviewed via `git diff data/seeds/`. Supports `--dry-run`, `--source global|<tenant-uuid>` (export the global taxonomy, or treat a template tenant as the source), `--out DIR`, `--no-backup`. Output is deterministic (sorted by natural key, fixed field order, only meaningful fields, regenerated metadata) so re-exporting an unchanged DB is a git no-op. 16 real-DB tests: per-file round-trip preservation against the shipped seeds, cross-export determinism, scoped export→re-ingest idempotency, the lossless-field guard, the backup/atomic pipeline, and the CLI surface.
- **Lossless seed read-side: anatomy `class_concept_slug` + biomarker concept-edge endpoints.** The seed ingest now reads `class_concept_slug` on `anatomy_structures.json` (preferred over the legacy ~8-value uppercase `category` enum), so anatomy whose class concept is outside the legacy set (e.g. a niche "lymphatic-structure" class) round-trips instead of silently falling back to "other-anatomy". The concept-edge ingest resolves a `biomarker` endpoint type by `BiomarkerDefinition.slug`, unblocking biomarker-level edges beyond panel membership. Both are backward compatible (legacy-only seed items keep working).
- **Backup export/import now carries the taxonomy + anatomy.** ZIP backups (`ExportType.FULL_BACKUP`) include three new sidecars so tenant-private customizations travel with the backup and survive a restore: `concepts.json` (tenant-scoped concepts + their multi-kind tags), `anatomy.json` (tenant-scoped body-part structures + the relations between them), and `concept_edges.json` (the polymorphic knowledge-graph edges — specialty→organ, biomarker→panel `MEMBER_OF`, AI-approved relationships). Import restores them in dependency order (concepts → anatomy → biomarkers/exams → edges) with UUID id-remapping: FK-bearing rows (examination `category_id`, biomarker `class_concept_id`, anatomy `class_concept_id`, edge endpoints) relink to the target's rows. References to global/seeded concepts that were deliberately not exported resolve by slug fallback against the target's re-seeded taxonomy. Edges with an unresolvable dangling endpoint are skipped with a warning rather than aborting the import. Restores are idempotent (upsert by slug / by natural key) so re-importing the same backup is a no-op. The `gather_anatomy` export is strictly tenant-scoped (`tenant_id == tid`) — it never leaks another tenant's `is_custom` anatomy. 17 new real-DB round-trip tests in `tests/test_export_import_taxonomy.py`.
- **Multi-kind concepts (taxonomy consolidation, part 2).** A concept can now carry multiple domain tags — "Blood Laboratory" is simultaneously an `examination_category`, a `biomarker_class`, and a `document_category` — instead of existing as 3 unrelated rows. The single-valued `concepts.kind` column is replaced by a many-to-many `concept_kind_tags` join table (reusing the existing `conceptkind` PG enum) plus a denormalized `primary_kind` column for cheap single-badge rendering. Migration `9a3f7c2e1b4d` is a pure schema change (create join table + `primary_kind`, backfill both from the legacy `kind`, swap the unique index from `(kind, slug, tenant)` → `(slug, tenant)`, drop `kind`); the 16 legacy duplicate concepts are de-duplicated in the seed data (`concepts.json`: 68 → 54 items), which is the single source of truth — the migration assumes a greenfield/empty concepts table (per the prior `consolidate_categories` note) and carries no hardcoded merge table. The new `concepts_with_kind()` helper replaces every legacy `Concept.kind ==` filter across 10 consumer files (services, AI pipeline, endpoints); `resolve_concept_by_slug` keeps its `kind` param (now an optional join-table filter) so resolvers needed no caller changes; `create_concept` accepts `kinds: List[ConceptKind]` (legacy single `kind` still wrapped for backward compat). Pydantic schema emits `kinds: List[str]` + `primary_kind`; the create endpoint accepts either `kinds: [...]` or legacy `kind`. `biomarker_category_to_concept_slug` drops the now-obsolete `-class` suffix. 6 new multi-kind model tests + updated concept/service/API/seed/taxonomy-tool tests; the dedup also fixed 3 previously-failing seed idempotency tests. Frontend (types, TaxonomyManager, ConceptGraphView) lands next.
- **Biomarker catalog auto-seeding.** `seed_default_catalog()` loads `data/seeds/default_catalog.json` (units + biomarker definitions) at startup via `CatalogImportService`, resolving the empty-catalog issue. Runs AFTER `seed_concepts` so `biomarker_class` concepts exist for `class_concept_id` resolution. **Biomarker panel memberships** (`seed_biomarker_panels()`) creates `MEMBER_OF` edges from biomarkers to panel concepts (e.g. LDL → Lipid Panel) from `data/seeds/biomarker_panels.json`. Both wired into `main.py` startup.
- **FHIR observation category on concept meta_data.** The `_BIOMARKER_CONCEPT_TO_FHIR_CATEGORY` and `_BIO_FALLBACK_FHIR_CATEGORY` hardcoded dicts in `persistence.py` are replaced by `meta_data["fhir_observation_category"]` stored directly on each `biomarker_class` concept in `concepts.json`. The FHIR mapping is now editable via the TaxonomyManager UI and lives with the concept (single source of truth). The `_fhir_observation_category()` function takes the concept object directly.
- **Unified taxonomy — Phase 7 (frontend).** Management UI and reusable components for the concept layer: `TaxonomyManager.tsx` — generic CRUD page with a domain switcher (all 16 ConceptKind values), inline editor with color/icon/alias/coding-system fields, icon picker from the Lucide catalog; `<TaxonomyTypeahead>` — server-backed debounced search component for picking concepts in forms (replaces free-text inputs); `conceptService.ts` — typed API wrapper for all /concepts + /concept-edges endpoints; `types/concept.ts` — full TypeScript types. Route: `/admin/system/taxonomy` (SYSTEM_ADMIN); `/examinations/categories` now renders TaxonomyManager. Sidebar item under Administration. Broken `examinationService` + `clinicalEventService` category functions repointed from the dropped endpoints to `/concepts?kind=examination_category` and `/concepts?kind=event_category`. TypeScript build + ESLint clean.
- **Unified taxonomy — Phase 6 (AI chatbot tool).** New `taxonomy` domain tool module auto-wired into the agentic chatbot via `@register_chat_tool("taxonomy")`: `search_concepts(search_term, kind)` — ranked trigram + alias search over the concept layer; `get_concept_neighborhood(concept_id, relation)` — one-hop graph traversal; `get_entity_concepts(entity_type, entity_id, relation)` — "what panels is this biomarker in?" / "what specialty does this doctor have?". All tools are live-read-only (AI never writes; proposals go through HITL). 4 tool tests including alias matching and error handling.
- **Unified taxonomy — Phase 5 (consolidation).** Drops the 4 old scattered category/group tables (`examination_categories`, `clinical_event_categories`, `biomarker_groups`, `biomarker_group_members`) and consolidates all category vocabulary into the unified `concepts` table. Migration `2b4ee2367046`: adds `class_concept_id` FK to `biomarker_definitions` + `anatomy_structures`, `specialty_concept_id` to `doctors`, `category_concept_id` to `documents`; repoints `examinations.category_id` + `clinical_event_types.category_id` FKs to `concepts.id`; drops old free-text/enum `category`/`specialty` columns; drops old tables. All 17 consumer files updated: `analytics_service` (clinical_groups from concept_edges), AI pipeline (`resolve_category` → concepts, form fillers), endpoints (examinations/clinical_events/biomarkers), schemas, export/import, seed service. Deleted `ExaminationCategory` model + endpoint + schema. 101 tests passing.
- **Unified taxonomy — Phase 4 (seeds).** Curated seed data for the concept layer: ~60 concepts covering 9 domains (specialties, body systems, examination categories, event categories, biomarker classes, biomarker panels, anatomy classes, document categories, ATC medication classes) + ~19 concept↔concept edges (specialty EXAMINES body_system, specialty PERFORMS examination_category, ATC class hierarchies via PART_OF). Idempotent upsert by `(kind, slug)` for concepts and `(src, dst, relation)` for edges; parent_slug resolution for hierarchical concepts. Integrated into startup seeding (`main.py` lifespan). 4 seed idempotency tests.
- **Unified taxonomy — Phase 3 (API endpoints).** REST endpoints for the concept layer: `GET/POST /concepts`, `GET/PUT/DELETE /concepts/{id}`, `GET /concepts/search?q=&kind=`, `GET /concepts/{id}/neighbors` (one-hop graph traversal), `GET/POST/DELETE /concept-edges`. Full RBAC enforcement (SYSTEM_ADMIN for global rows; ADMIN for tenant-scoped; USER read-only), tenancy filtering, and validation. Pydantic v2 schemas (`ConceptCreate/Update/Response`, `ConceptEdgeCreate/Response`, `NeighborResponse`). 11 HTTP integration tests covering CRUD, search, edge creation, neighbor traversal, RBAC (USER 403 on write, USER 200 on read), and validation errors.
- **Unified taxonomy — Phase 2 (service layer + search).** `ConceptService` provides full CRUD for concepts + edges with RBAC enforcement (SYSTEM_ADMIN for global rows; ADMIN for tenant-scoped; USER read-only), tenancy filtering (`or_(tenant_id == tid, tenant_id.is_(None))`), soft-delete-with-retire-if-referenced lifecycle, one-hop neighbor traversal, and entity→concept lookup ("what panels is this biomarker in?"). AI-proposed edges land as `status=PROPOSED` and are invisible to graph queries until approved. `search_concepts()` added to `catalog_search_service` — trigram + ilike + JSONB aliases containment, optionally filtered by `kind`, excludes retired/deleted. 20 real-DB integration tests covering CRUD, RBAC, tenancy isolation, edge validation, neighbor traversal, proposed-edge lifecycle, and ranked search.
- **Unified taxonomy / knowledge-graph foundation (Phase 1).** Introduces two new tables that form the backbone of a consolidated medical vocabulary system: `concepts` (controlled-vocabulary nodes for specialties, examination/event/biomarker/anatomy/document categories, biomarker panels, medication classes, diseases, body systems, etc.) and `concept_edges` (typed polymorphic edges between concepts and/or domain entities — `MEMBER_OF`, `EXAMINES`, `PERFORMS`, `TREATS`, …). This replaces the scattered `examination_categories`, `clinical_event_categories`, `biomarker_groups` tables and free-text `category` columns (consolidation migration follows in a later phase). Six new enums (`ConceptKind`, `ConceptStatus`, `ConceptProvenance`, `EdgeApprovalStatus`, `EdgeEndpointType`, `ConceptRelationType`). `coding_system` is a free string (not enum) to support open-ended terminology systems (LOINC, SNOMED, ATC, ICD-10, CVX, MeSH). Trigram GIN index on `name`+`slug` powers ranked search; JSONB GIN on `aliases` enables containment matching. Unique partial indexes use `COALESCE(tenant_id, sentinel)` to correctly handle global (NULL-tenant) rows. Migration `1a3dd1256035`.

### Fixed
- **Export/import jobs stuck in "Processing" forever (completed_at type bug).** `ExportService.complete_job`/`fail_job` and `ImportService._complete_job`/`_fail_job` passed `datetime.now(timezone.utc).isoformat()` — a **string** — into the `completed_at` column (`TIMESTAMP WITH TIME ZONE`). asyncpg rejects strings (it needs a real `datetime`), so the export/import actually succeeded (the ZIP was written) but the DB update marking the job `COMPLETED` crashed, and the fallback `fail_job` crashed the *same way* — leaving the job stuck in `PROCESSING` permanently and the UI never showed an error. Now passes the `datetime` object directly. 3 new real-DB regression tests cover `complete_job`/`fail_job` (the mocked export tests had skipped these, which is why it slipped through).
- **Biomarker class link (`class_concept_id`) was silently dropped on every backup round-trip.** The export emitted the concept *name* (e.g. `"Blood Laboratory"`) in the legacy `category` field, but the import's `biomarker_category_to_concept_slug` only swaps `_`→`-`, turning that name into `"blood laboratory"` (with spaces) which never matched the real slug `"blood-laboratory"` — so `class_concept_id` resolved to `None`. The export now also emits `class_concept_slug` (the canonical concept slug) and `CatalogImportService` resolves that ahead of the legacy name→slug path. The ontology-catalog-URL import (whose JSON uses the underscore convention `blood_laboratory`) is unaffected.
- **Examination category (`category_id`) was exported but never restored.** The `examinations.json` sidecar carried `category_id` + `category_details`, but `_restore_examinations` ignored both — imported exams always came back uncategorized. It now relinks the category (via the concept id-remap, with slug fallback to the target's re-seeded global taxonomy) and also restores the previously-dropped `organization_id`, `source_integration_id`, `external_id`, and `auto_extract_metadata` fields.

Existing rules can be enabled/disabled, edited, deleted, and test-fired. The builder is a new "Biomarker Rules" tab in the Notification Center. Separately, the misleadingly-named `/alerts` page (which actually showed allergies) is renamed `ClinicalAlerts` → `AllergyAlerts`, with the sidebar label updated to "Allergy Alerts" — freeing the "alerts" name for the biomarker-alert system. i18n keys added for `en`/`el`.
- **Unified notification system — Phase 2 (real-time delivery + user-scoped inbox).** The notification bell is now real-time and user-scoped: a new `/ws/notifications` WebSocket streams each user's notifications from a per-user Redis channel (`user:{id}:notifications`) authenticated via the `["bearer", token]` Sec-WebSocket-Protocol subprotocol, replacing the previous 30-second patient-scoped poll. The bell now works **without a patient selected** (so system/tenant notifications surface), renders severity dots, action buttons (deep-link to the relevant resource), and category icons for all notification types (reminder/alert/hitl/agent/system/integration/clinical_event), and shows a live/reconnecting indicator. The Notification Center page (`/notifications`) is rebuilt with Inbox / All / Reminders tabs, category + status filters, mark-all-read, and dismiss. The frontend `notificationService` and `notificationStore` are rewritten around the Phase 1 fan-out model (inbox items, optimistic read/dismiss, live-event handler). A new `useNotificationStream` hook manages the WebSocket lifecycle (auto-reconnect with backoff + fallback unread-count poll if the socket can't open). The service worker's obsolete `/notifications/{id}/delivered` PATCH is removed (delivery status is now tracked server-side by the push delivery worker). i18n keys added for `en`/`el`.
- **Unified notification system — Phase 1 backend (fan-out model).** The notification subsystem is rebuilt around a 3-table fan-out architecture: one immutable `Notification` event (with `source`/`category`/`severity`/`type`, nullable `patient_id`/`tenant_id` for system/tenant broadcasts), `NotificationRecipient` (per-user inbox state: unread/read/dismissed), and `NotificationDelivery` (per-recipient per-channel delivery log). A single `notification_service.emit()` entry point now produces every notification — it resolves target specs (`USER`/`PATIENT`/`DOCTOR`/`TENANT`/`SYSTEM`) into concrete user ids, fans out recipients + deliveries, optionally links a FHIR `Communication` for clinical sources, publishes to a per-user Redis channel, and enqueues push delivery. The orphaned `AlertModel` is replaced by `NotificationRule` (operator + value + biomarker + multi-recipient targets), evaluated **event-driven** on observation create (`fhir_service.create_observation` → `notification_rule_service.evaluate_and_fire`) with per-rule cooldown. New endpoints: `/notifications/inbox`, `/unread-count`, `/{id}/read|dismiss`, `/read-all`, `/notifications/admin` (tenant/cross-tenant feed), and `/notification-rules` CRUD + `/test`. The `deliver_notification` worker task now iterates per-recipient deliveries (dead-subscription pruning preserved). The legacy `/alerts` endpoints, `AlertModel`, and the dead `biomarker_update` event trigger are removed. Schema folded into the squashed `0001` migration (no backfill — greenfield). **Frontend (real-time bell, Notification Center, rule builder) lands in Phase 2.**
- **FHIR facade tenant bridge: service-account auth + X-Tenant override (F19, G10).** External systems can now connect to the FHIR facade (`/fhir/R4/*`) with a service-account JWT (minted by admins via `POST /auth/service-account`) instead of a user password. The CapabilityStatement documents both auth modes. `SYSTEM_ADMIN` can scope into any tenant via the `X-Tenant: <uuid>` header. Provenance records the external system identity (`client_id`) as a third agent entry on service-account-initiated writes. This is the stepping-stone before full SMART-on-FHIR (F1) — deferred to Stage 4.
- **Service-account JWT auth for external systems (F19).** New `POST /auth/service-account` endpoint (ADMIN/MANAGER/SYSTEM_ADMIN only) creates a password-less machine account (`is_service_account=True`) and returns a long-lived JWT carrying `is_service_account=True` + `client_id=<instance_name>`. The token works as a Bearer against the FHIR facade (`/fhir/R4/*`) and the REST API. Service accounts are bound to one tenant (the JWT's `tenant_id`). `TokenData` gained `is_service_account` + `client_id` fields. `/auth/login` now rejects service-account emails (403). `UserModel.hashed_password` is now nullable (service accounts have no password). Migration `f19a2b3c4d5e` adds the `is_service_account` column with a partial index. This is the tenant-bridge stepping-stone before full SMART-on-FHIR (F1) — external systems can now connect with a bearer token instead of a user password.
- **Push resilience: per-row 401-race retry, cursor integrity, batch isolation.** Three robustness fixes: (1) Push now retries once after a 401 race (token expired between the `get_live_token` check and the PUT) via `force_refresh` — mirroring the pull path's `_authorized_search` pattern. Previously a 401 mid-batch aborted the entire push. (2) The `last_pushed_at` cursor now only advances past *successfully* pushed rows (tracks the max `updated_at` among created/updated outcomes). If ALL rows fail, the cursor stays unchanged → full retry next cycle. Previously the cursor advanced to `now` unconditionally, so transiently-failed rows were never retried (silent data loss). (3) Per-row error isolation — one bad row counts as `error` and the batch continues; only `insufficient_scope` (H1) aborts the batch early.
- **Outbound push now POSTs a Provenance to the remote server (H3).** Hospitals typically require this for regulatory audit ("Health Assistant pushed this from integration X"). After each successful push (PUT 201/200), `_push_one` POSTs a Provenance with the remote Observation id as the target, the local Device as the agent, and a CREATE activity code. Best-effort: Provenance failure (server doesn't support it, network error) is logged and never aborts the push. The push result carries `provenance_created`/`provenance_failed` counters. New `fhir_create` (POST) primitive added to `sdk/fhir.py` alongside the existing `fhir_conditional_update` (PUT).
- **Outbound push requests write scope when push is enabled (H1).** `DEFAULT_SCOPES` was read-only (`patient/*.read`); a spec-strict SMART server rejects `PUT /Observation` with 403. New `PUSH_SCOPES` adds `patient/*.write`; `begin_oauth` passes `push_enabled` when `sync_direction` is `both` or `push_only`. The config-flow description documents that switching to push requires re-authorizing. 403 `insufficient_scope` is now detected — the push stops and the result carries an actionable "re-authorize with write permissions" warning instead of silently failing every row.
- **Error messages now include parsed FHIR OperationOutcome diagnostics.** `parse_operation_outcome` existed but was never called on any error path — all integration SDK raises used raw `response.text[:300]`, making hospital 422/400 rejections hard to debug. Both `fhir_conditional_update` and `http_request` now extract `OperationOutcome.issue[].diagnostics` (the most useful field — e.g. "code system http://loinc.org not recognized") before falling back to the raw body.
- **Outbound push: multi-component observations (blood pressure) + multi-range referenceRange + note preservation.** `fhir_observation_to_create` previously returned `None` for any observation lacking both `valueQuantity` and `valueString` — silently dropping blood pressure (the most common vital, which uses `component[]` for systolic + diastolic) and panels (BMP, CBC). The early-return now checks for `component[]` — these observations survive and their components are mapped onto the `Observation.component` JSONB column (added in Phase 6). `note[0].text` is mapped to the ORM `comment` field. Additionally, `_extract_reference_range` (renamed from `_convert_reference_range`) now returns the full canonical `referenceRange[]` list instead of flattening only the first range to `{min, max}` — multi-range observations (age-stratified, gender-specific) preserve all ranges with `type`/`appliesTo`/`text`/units intact.
- **OAuth hygiene: atomic state consume, 429 fix, token revocation, DCR guard.** Four robustness fixes in the integration SDK auth layer: (1) `OAuthStateStore.consume` now uses `GETDEL` (Redis ≥ 6.2) for a truly atomic read-and-delete — the old GET+DELETE had a TOCTOU race where duplicate OAuth callbacks could consume the same state. (2) The token-endpoint 429 response is now correctly classified as `IntegrationRateLimitError` (was `IntegrationAuthError` — rate-limited token endpoints would pause the integration instead of retrying). (3) `SmartOAuth.revoke(integration)` calls the SMART `revocation_endpoint` (RFC 7009) on disconnect — `DELETE /instance/{id}` now revokes tokens before deleting, preventing stale tokens on hospital servers. Best-effort: network errors are swallowed. (4) If a SMART server advertises a `registration_endpoint` but DCR returns no `client_id`, the flow now raises an actionable `IntegrationAuthError` instead of silently producing an authorize URL with an empty `client_id`.
- **Integration sync pipeline consolidated into shared `run_sync` helper.** The pull → convert → biomarker-map → telemetry-split → push → log pipeline was previously copy-pasted across 4 call sites (the Celery worker, the manual-sync endpoint, the webhook path, and the bridge provider). It now lives in a single `IntegrationSyncService.run_sync(integration, db, provider, *, source)` method that both the worker and the manual-sync endpoint delegate to. The shared helper also acquires the Redis dedup lock internally — fixing a race condition where the manual-sync endpoint didn't respect the worker's lock (concurrent Sync Now + beat could double-write). The 3-tier error contract (`IntegrationAuthError` → status=ERROR; `IntegrationRateLimitError` → retry; generic → log) is now uniform across both sites. The manual-sync endpoint returns HTTP 409 when a sync is already in progress (lock held), 401 on auth failure, 429 on rate limit, 500 on other errors.
- **`Observation.component` JSONB column added (prereq for H2 push).** The FHIR R4 `Observation.component` (`0..*`) — used by multi-component observations like blood pressure (systolic + diastolic) and panels (BMP, CBC) — is now a first-class JSONB column on the ORM model (migration `i7b8c9d0e1f2`, with a GIN index for future `component-code-value-*` search). The import converter (`fhir_to_observation_orm`), `_upsert_observation` (create + update paths), `to_fhir_dict()`, and `to_dict()` all carry it. The `ObservationCreate`/`Update`/`Response` schemas accept `component: Optional[List[Dict]]`. ### Fixed
- **Celery periodic tasks crashed intermittently with `RuntimeError: Event loop is closed` / `Future attached to a different loop`.** Root cause: each task's `get_async_session()` created a fresh `AsyncEngine` with its own connection pool, and the `@async_task` decorator created a fresh `asyncio` event loop per task. asyncpg connection protocol transports bound to a previous task's loop resurfaced in subsequent tasks via SQLAlchemy's pool, hitting a closed loop. - **`normalized_value` had two bugs in the OCR persistence path.** (1) Skip-condition: it was only computed when the biomarker had a preferred unit *different* from the matched raw unit; auto-created biomarkers (no preferred unit) kept raw_value, mixing units across labs in trend history. - **Web Push dead subscriptions accumulated forever.** `webpush_service.send_web_push` swallowed every `WebPushException` including the 410 Gone / 404 Not Found cases that RFC 8030 specifies as "subscription permanently dead", returning `False`. ### Fixed

### Fixed (FHIR conformance — runtime)
- **SDK-built Observations no longer silently dropped by FHIR validation.** `ObservationBuilder.build()` stripped tzinfo "for asyncpg compat" (asyncpg handles tz-aware natively), so `isoformat()` produced e.g. `'2026-06-20T22:39:56.471381'` which failed the FHIR R4 regex. ### Fixed (legacy)
- **HITL tasks lost on stream interruption**: tasks were only saved at the END of the `_chat_stream` generator. If the stream was interrupted (LLM error, client disconnect) after the `[HITL_TASK]` SSE chunk but before completion, the task was never persisted — breaking `/resolve` (404) and `/resume`. Tasks are now **proactively saved** the moment they're detected; `update_message_fields` patches the final content when the stream completes normally.
- **`"object async_generator can't be used in 'await' expression"`**: `resume_after_hitl` is an async generator (uses `yield`) — the `/resume` endpoint was incorrectly awaiting it. Fixed: `async for chunk in service.resume_after_hitl(...)` (no await).

'address' Input should be a valid dictionary ... input: [{...}]`): root-caused to a Pydantic schema / FHIR cardinality mismatch (the schema typed `address` single while stored JSONB + FHIR use a list). ### Operational notes for deploy
1. **Set `INTEGRATION_SECRET_KEY`** (Fernet key) in `.env`. Generate one with:
   ```bash
   python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
   ```
2. **Run the api_key backfill** after deploying:
   ```bash
   cd backend && PYTHONPATH=. python scripts/encrypt_existing_api_keys.py --dry-run
   cd backend && PYTHONPATH=. python scripts/encrypt_existing_api_keys.py
   ```
3. **Run migration `f1a2b3c4d5e6`** (`alembic upgrade head`) — creates the previously-dropped `idx_ai_models_provider_active` index.

5. **Set `POSTGRES_PASSWORD`** explicitly in `.env` — the insecure `admin123` default was removed. Production boot will refuse known-weak values.
6. **Webhook HMAC**: integrations that want HMAC verification should set `user_config["webhook_secret"]` to a strong random string. The sender must compute `HMAC-SHA256(secret, raw_body)` and send the hex digest in the `X-Webhook-Signature` header.
7. **WebSocket frontend migration**: update the frontend `useWebSocket` hook to connect with `new WebSocket(url, ["bearer", token])` so the token travels via the subprotocol instead of the query string. The query-string fallback remains for backward compatibility.
8. **AuditLog**: the `audit_logs` table is now populated for FHIR create/delete operations. Query it for provenance: `SELECT * FROM audit_logs WHERE resource_type = 'Observation' ORDER BY created_at DESC`.
