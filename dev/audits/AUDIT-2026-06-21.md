# Health Assistant — Full Project Audit (2026-06-21)

**Scope:** Full scan of backend, frontend, AI pipeline, clinical-data layer, and integrations framework/SDK.
**Baseline version:** `0.2.1-rc.2`
**Method:** Parallel deep-dive exploration agents (read-only) across all five domains, cross-referenced against `docs/STATUS.md`, `docs/DEVELOPMENT_PLAN.md`, and the project-overview skill.
**Purpose:** Establish a tracked baseline of discrepancies, mis-alignments, missing features, and FHIR-compatibility gaps before starting the v0.3.0 work-stream.

---

## Severity rating

| Level   | Meaning                                                                              |
|---------|--------------------------------------------------------------------------------------|
| **C**   | Critical — runtime crash, data corruption, or security exposure. Ship-blocking.      |
| **H**   | High — incorrect behaviour, missing core capability, or significant hygiene risk.    |
| **M**   | Medium — works but degrades quality/security/UX; should fix before stable.            |
| **L**   | Low — polish, drift, or aspirational roadmap item.                                   |

Each item carries a status checkbox (`[ ]` open, `[x]` fixed). Items are referenced by absolute `file:line` where possible.

---

## Executive summary

- **13 critical/high runtime bugs** found, including cross-patient data exposure on the FHIR Observation list endpoint, a 60-second background task that writes telemetry data into the wrong store, and a silently-dropped DB index caused by a `__table_args` typo.
- **11 security concerns** above the hygiene baseline, dominated by plaintext + API-exposed AI provider keys, missing tenant scoping on telemetry endpoints, and an information-leaking global exception handler.
- **FHIR compatibility is "FHIR-ish", not a conformant R4 REST server**: no `/metadata` CapabilityStatement, no Bundle responses on search, missing Condition/Encounter/Device/Provenance resources, SDK only produces Observation.
- **SDK/webhook parsers emit invalid LOINC codes and non-UCUM units** for non-clinical concepts (distance, calories, sleep stages).
- **Test coverage is sparse in exactly the areas where bugs hide**: `sync_active_integrations`, webhook parsers, `ObservationBuilder`, ai_assistance, notifications, alerts, admin surface.
- **Frontend** has 483 `any` usages, 5 test files total, 61 missing Greek i18n keys, hardcoded `localhost:8000` in PWA runtime caches, and a GraphQL client that never refreshes its auth header.
- **All major consumer wearables** (Apple Health, Fitbit, Garmin, Withings, Whoop, Oura, Libre/Dexcom) and **HL7 v2 / FHIR Subscriptions / CDA** are absent — the architecture is ready (SMART SDK + bridge + webhook parsers) but no first-party providers exist.
- **Biomarker Insights & Correlations** (roadmap headline) is unimplemented; schema exists, no statistical computation.

## Section K — Strengths to preserve

These should not be regressed while addressing the items above:

1. **Tenant isolation is hard** — every query filters by `tenant_id`; USER role further restricted to `Patient.user_id`.
2. **FHIR write-time validation** via `assert_valid_fhir()` chokepoint — invalid FHIR never persists.
3. **HITL security model** — every clinical write goes through `propose_*` → user review → canonical REST endpoint. AI never writes directly (except `update_examination_notes`).
4. **SDK OAuth2 + SMART foundation** — PKCE + DCR + Redis state + refresh-on-401 is production-grade.
5. **TimescaleDB telemetry split** — frequency-based routing with OHLC aggregation concept (when §A.8 fixed).
6. **Resilience patterns** — `DATABASE_AVAILABLE` flag, fresh engine per Celery task, stuck-exam cleanup.
7. **Centralized versioning** via `scripts/version_manager.py`.
8. **Export/Import system** — FHIR R4B Bundle + BagIt-style ZIP, validated imports with SHA256 manifest + cross-tenant id remapping.
9. **pg_trgm typo-tolerant catalog search** across all catalogs.
10. **Metadata-driven Clinical Events** — frontend dynamically renders JSONB schemas.
