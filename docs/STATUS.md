# Health Assistant — Project Status & Roadmap

What's implemented, what's in progress, and what's planned for the self-hosted, open-source health records platform.

## Current Status

**Backend**: Running on http://localhost:8000  
**Frontend**: Running on http://localhost:3000  
**API Docs**: http://localhost:8000/docs

## Implementation Progress

### ✅ Completed

#### Backend
- FastAPI server with async support (Python 3.12+)
- SQLAlchemy 2.0 ORM with PostgreSQL & Alembic migrations
- JWT authentication with refresh tokens and presigned download tokens
- FHIR resource models (Patient, Observation, DiagnosticReport, Medication)
- Full Clinical Visit system (Examinations & Doctors)
- AI OCR & NLP Pipeline (OpenAI Vision/LLM + spaCy)
- Background task processing via Celery & Redis
- Modular Notification Framework with Web Push (VAPID) support — replaced with the **unified notification system** (multi-source, multi-recipient, multi-channel, role-aware, real-time via WebSocket). See [NOTIFICATION_SYSTEM.md](NOTIFICATION_SYSTEM.md).
- Unit converter service
- Anomaly detector service (Reference range based)
- Medication interactor service
- Centralized semantic versioning manager
- **Unified taxonomy & knowledge graph** — multi-kind `concepts` (one term can carry several domain tags via `concept_kind_tags`) + typed polymorphic `concept_edges`, replacing the scattered category tables. Searchable concept CRUD in the Catalogs workspace at `/catalogs?type=concept` (TaxonomyManager retired), REST API at `/concepts` + `/concept-edges`. See [TAXONOMY.md](TAXONOMY.md).
- **Unified Catalog Registry + cross-domain knowledge graph** — a thin registry-driven `/catalogs` meta-layer gives every clinical catalog (anatomy, taxonomy, biomarkers, medications, allergies, **vaccines**) one CRUD/RBAC/search/FHIR/edge contract (`app/catalogs/`). `concept_edges` is the **single** link system: biomarker→anatomy `AFFECTS`, biomarker↔event-type `MONITORS`, biomarker→panel `MEMBER_OF`, specialty→organ `EXAMINES`, etc. Cross-catalog traversal at `GET /catalogs/{type}/{id}/relations` (recursive CTE); unified search at `GET /catalogs/search` + the global `/search` (both cover all registered catalogs). **Vaccinations (Phase 5)** are implemented end-to-end: `VaccineCatalog` + `PatientImmunization` tables, `/vaccines/*` REST CRUD, FHIR R4 `Immunization` facade resource, CVX-coded seeds. Phases 0–5 of `dev/plans/unified-catalog-architecture-2026-07-08.md` landed; diseases-as-concepts (Phase 6), the admin workspace (Phase 7), and LLM tools (Phase 8) landed. The legacy `biomarker_relationships` + `biomarker_event_correlations` tables are dropped (migrated to edges).
- **Export & Import (backup) system** — FHIR R4B Bundle + BagIt-style ZIP exports at patient/group/system scope; validated imports with SHA256 manifest verification and cross-tenant id remapping (see [EXPORT_IMPORT.md](EXPORT_IMPORT.md)). Admin-only UI at `/settings/export-import` (export form, drag-and-drop restore, live job polling, download).
- **Agentic AI Copilot + human-in-the-loop (HITL) proposals** — the chat assistant proposes clinical write actions (create a clinical event, add a biomarker to an examination, add a medication, define a new biomarker/medication in the catalog); the user reviews/edits and explicitly confirms before anything is saved (the AI never writes directly). After resolution, the agent gets an **auto-resume continuation turn** with structured outcomes fed back. **Parallel proposals** (multiple independent actions per turn) and a **Continue button** for partial resumes are supported. See [AI_SYSTEM.md §4.1](AI_SYSTEM.md).

- **Integrations SDK follow-ups pass** (workstreams A, D, B, E; see [CHANGELOG.md](../CHANGELOG.md) for per-commit detail). Two phases of work extending the Integrations Framework's SDK and capability surface; baseline plan at `dev/plans/integrations-sdk-followups-2026-07-21.md`. **~60 new tests across 13 files, 280+ integration tests green.**
  - **A — SDK robustness pass.** Extracted a single `_retry_request(do_request, *, url, method, max_retries)` primitive in `integrations/sdk/http.py` that owns the SDK's full retry/backoff/status-mapping contract — `http_request`, `fhir_create`, `fhir_conditional_update`, and `_request_json` (OAuth/DCR/token-refresh) all delegate to it. Backoff is now **full-jitter** (AWS-recommended) so concurrent clients no longer retry in lockstep; `paginate_bundle` and `fhir_search` default to `max_pages=100` (was unbounded); vestigial `BaseHealthProvider.fetch_json` (zero call sites) removed. OAuth/DCR/token-refresh calls are retry-aware for the first time.
  - **D — Service-context actor resolver.** `app/services/integration_actor.py` exposes `resolve_integration_actor(db, integration) -> TokenData`, which lets Celery / webhook / api-proxy contexts call the same `current_user`-shaped service functions interactive UI requests use. The integration inherits its owner's tenant, role, and user_id — same RBAC, same audit provenance, same tenant scoping. Foundational for B and E.
  - **B — Clinical Events opt-in hook.** Providers can pull longitudinal events (FHIR `Condition` / `EpisodeOfCare`) by overriding `supports_clinical_events` + `pull_clinical_events`. Migration `b1l2i3n4t5ev` adds `source_integration_id` + `external_id` columns + a partial unique index for dedup at the DB layer. The engine resolves the actor, calls `clinical_event_service.create_event`, and dedups when the provider sets `external_id`.
  - **E — Examinations opt-in hook + service chokepoint + bridge migration.** New `app/services/examination_service.py` exposes `create_examination` as the canonical write path (the `POST /examinations` endpoint was inlined; integrations had no clean call site). Migration `e1x2a3m4i5n6` adds the matching partial unique index. Providers opt in via `supports_examinations` + `pull_examinations`. The bridge provider (`integrations/health_assistant_bridge/`) was migrated to the service — **fixes a real silent-drop bug** where the bridge set `category_id=` (a column that no longer exists; renamed to `category_concept_id` when categories moved into the unified taxonomy), dropping the category on every exam it created.
- **Integrations SDK follow-ups pass — F + G + C** (see [CHANGELOG.md](../CHANGELOG.md) for per-commit detail). Three more workstreams extending the SDK capability surface on top of stack 1 (A, D, B, E); baseline plan at `dev/plans/integrations-sdk-followups-2026-07-21.md`. **~60 new tests across 7 files, 230+ integration tests green.**
  - **F — Catalog proposals opt-in hook.** Providers that discover new catalog entries upstream (a wearable adding a "Sleep Quality Score" metric, a hospital mapping disease↔biomarker, a medication subscription) opt in via `supports_catalog_proposals` + `pull_catalog_proposals`. The engine **auto-applies** each proposal through `catalog_proposal_service.apply_proposal`, which routes by `kind` (biomarker / medication / concept / edge) to the canonical write service and stamps `ConceptProvenance.INTEGRATION` on `ConceptEdge.source` + a `meta_data["_provenance"]` tag on biomarkers. Idempotent per kind (slug / name / endpoints). Closes the gap left by `ConceptProvenance.INTEGRATION` (declared in the enum but unwritten before F).
  - **G — HITL proposals for integrations.** The human-in-the-loop counterpart to F — same payload shapes, same write path on approve, but the platform **queues each proposal for human review** instead of auto-applying. Three opt-in hooks: `supports_hitl_proposals` / `pull_hitl_proposals` / `handle_proposal_resolution`. New `IntegrationProposal` model + migration `g1h2i3t4l5pr` (PROPOSED → CONFIRMED / DISMISSED / FAILED state machine, sha256-deduped). Three new endpoints (`GET/POST /api/v1/integrations/instance/{id}/proposals/...`). On approve, the resolver delegates to the same `apply_proposal` write path F uses — single catalog write path regardless of how the proposal entered the system. Frontend review card + modal (parallel to `HitlTaskCard`) on the roadmap; until then proposals are visible in the notification inbox + resolvable via REST.
  - **C — Document pull opt-in hook.** Providers that can deliver document bytes upstream (hospital portal pulling scanned lab reports, fax-to-email gateway, wearable companion app syncing ECG printouts) opt in via `supports_documents` + `pull_documents`. The engine routes each `DocumentPull` spec through `document_service.ingest_document_bytes` — extracted from `upload_document` in C.1 as the canonical bytes-shaped ingestion path the UI wrapper now delegates to. Same storage path, same DB shape, same OCR dispatch. Two per-sync caps (`INTEGRATION_MAX_DOCS_PER_SYNC = 20`, `INTEGRATION_MAX_DOC_BYTES_PER_SYNC = 50 MiB`). Document-level dedup landed in the follow-up improvements pass (see below).
- **Integrations SDK improvements pass** (5 items; plan at `dev/plans/integrations-sdk-improvements-2026-07-21.md`; CHANGELOG entry has the full per-item breakdown). Five independent improvements that surfaced while expanding `dev_dummy` into a reference implementation. **113 new tests; 586 integration-related tests green.**
  - **1 — Upstream `Retry-After` honoured via Redis cooldown.** `IntegrationRateLimitError.retry_after_seconds` carries the captured header value through `_retry_request`; `integration_sync_service` exposes `set_rate_limit_cooldown` / `is_rate_limited` / `clear_rate_limit_cooldown` that write a Redis key `sync_cooldown:{integration_id}` (TTL clamped [60s, 1h]); the worker skips the integration while the cooldown exists. Degrades gracefully when Redis is down.
  - **2 — Webhook HMAC helpers extracted to a reusable SDK module.** New `integrations/sdk/webhook_security.py` with `verify_hmac_signature` (constant-time; strips GitHub/Slack/Stripe `sha256=` / `sha256:` prefixes), `verify_canonical_signature` (the two-way-API canonical form `METHOD\npath\n[timestamp\n]body` with ±5min replay-skew window), and `get_signature_header` (case-insensitive lookup). The endpoint and provider `dev_dummy` both delegate; behaviour preserved.
  - **3 — Document-level dedup.** Migration `d1o2c3u4m5e6` adds `source_integration_id` + `external_id` columns + partial unique index `uq_document_integration_dedup` on `(tenant_id, patient_id, source_integration_id, external_id)` — mirrors the pattern on examinations + clinical_events. `ingest_document_bytes` gains matching kwargs; on dedup hit returns the existing row as-is (no re-file-write, no re-OCR). `DocumentPull.external_id` surfaces it in the SDK.
  - **4 — Notification digest collapsing.** Migration `n1o2t3i4f5y6` adds `dedup_key` + `dedup_expires_at` columns + a non-unique composite lookup index. `NotificationSpec.digest_key` + builder method; `emit(dedup_key=..., dedup_ttl_seconds=...)` collapses repeated emissions inside a TTL window (default 6h, clamped [60s, 7d], tunable via `settings.NOTIFICATION_DEFAULT_DIGEST_TTL_SECONDS`). No unique partial index — Postgres partial indexes can't reference `now()` in the predicate.
  - **5 — `setup(config)` gets real config.** The registry forwards `SystemIntegration.global_config` to `provider.setup(system_config)` instead of the legacy empty dict. Signature widened to `config: dict | None = None`. `dev_dummy` overrides `setup` as the reference example.
